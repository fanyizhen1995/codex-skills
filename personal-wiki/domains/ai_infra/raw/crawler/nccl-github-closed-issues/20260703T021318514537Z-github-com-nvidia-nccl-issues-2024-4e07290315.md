# Ring Broadcast hangs at >16M elements with cudaIpcMemLazyEnablePeerAccess on H200/NVSwitch

URL: https://github.com/NVIDIA/nccl/issues/2024
State: closed
Labels: 
Closed at: 2026-07-02T15:46:46Z
Merged at: 

## Environment
- 2x AWS P5en.48xlarge (8x H200 143GB, NVSwitch intra-node)
- 16 EFA per node, EKS Kubernetes
- NCCL 2.27.5, PyTorch 2.x, CUDA 12.x
- Processes: 16 separate processes via Ray (1 GPU per process)

## Problem

`torch.distributed.broadcast()` hangs for tensors >16M elements (64MB) when:
1. Separate processes use `cudaIpcOpenMemHandle(..., cudaIpcMemLazyEnablePeerAccess)`
2. On H200 with NVSwitch (does NOT happen on H100 without NVSwitch)
3. Using Ring algorithm (Broadcast always uses Ring in NCCL)

Smaller broadcasts (≤16M elements) work. AllReduce with Tree algorithm works at ALL sizes.

## Benchmark

| Operation | Size | Result |
|-----------|------|--------|
| Broadcast 1M | 4MB | **0.8ms** ✅ |
| Broadcast 16M | 64MB | **0.8ms** ✅ |
| **Broadcast 233M** | **933MB** | **HANG** ❌ |
| AllReduce 233M (Tree) | 933MB | **9.6ms** ✅ |
| torchrun Broadcast 233M | 933MB | **<5ms** ✅ |

## Root Cause Analysis

The Ring algorithm creates a circular dependency (Rank 0→1→...→15→0). With separate processes, intra-node P2P uses `cudaIpcOpenMemHandle` with `cudaIpcMemLazyEnablePeerAccess` (p2p.cc:299). The CUDA driver defers NVSwitch multicast configuration to first memory access.

When the Ring kernel executes, ALL ranks simultaneously trigger the lazy init from GPU kernels (`prims_simple.h:waitPeer`). The NVSwitch driver serializes these through a global lock, and the Ring's circular dependency creates a classic deadlock.

**With torchrun**: All GPUs visible per process → `cudaDeviceEnablePeerAccess()` called eagerly (p2p.cc:329) → no lazy init → no deadlock.

**With separate processes (Ray)**: Only 1 GPU visible → `busIdToCudaDev()` returns -1 → falls through to IPC path → lazy init → deadlock.

## Patches Attempted (all fix ≤16M but NOT 233M)

1. `cudaMemcpy(dummy, *devMemPtr, 1, D2H)` after IPC open — fixes ≤16M
2. `cudaMemset(*devMemPtr, 0, 4096) + cudaDeviceSynchronize()` — same result
3. Both main IPC (line 299) and proxy IPC (line 1056) patched — same result

The 233M broadcast requires multiple Ring channels simultaneously. The eager touch forces init for individual IPC handles but the multi-channel concurrent Ring kernel still triggers the NVSwitch driver deadlock.

## Why Tree Works But Ring Doesn't

Tree has an acyclic dependency graph — always a leaf that can complete without waiting downstream. Ring's circular dependency means ALL ranks must make progress simultaneously, which deadlocks when the NVSwitch driver serializes lazy IPC init.
