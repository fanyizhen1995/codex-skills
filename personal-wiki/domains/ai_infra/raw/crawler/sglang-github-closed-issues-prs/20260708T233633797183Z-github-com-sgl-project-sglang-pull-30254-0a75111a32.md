# [Fix] PD-disaggregation mooncake RDMA Transport KV data race

URL: https://github.com/sgl-project/sglang/pull/30254
State: closed
Labels: 
Closed at: 2026-07-08T06:40:43Z
Merged at: 


## Motivation

In PD disaggregation scenario, when using Mooncake-RDMA as kv-cache transport method,  there is a data race between the CUDA compute stream that writes the KV cache and the RDMA NIC that reads it.

Because batch_transfer_sync_write runs on the CPU thread (not on any CUDA stream) and the RDMA NIC accesses GPU memory via PCIe DMA, the NIC does not participate in CUDA stream ordering. If the forward kernels writing this KV chunk have not yet completed on the compute stream when the RDMA op is issued, the NIC can read stale or partially-written bytes and transmit them to the decode side.




## Modifications

   Record a CUDA event on the current compute stream when a chunk is enqueued, and synchronize on that event in the transfer worker before issuing the RDMA op:
   - add_transfer_request (scheduler thread): create a torch.cuda.Event and event.record() on the current stream. This is non-blocking for the scheduler.
   - transfer_worker (background thread): after popping the chunk, call event.synchronize() before batch_transfer_sync_write. The worker blocks only until the specific kernels writing this chunk
   have finished, not the whole stream.
   - TransferKVChunk: carry the event as an optional field so the fix is a no-op on non-CUDA backends.
