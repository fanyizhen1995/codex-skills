### Describe the bug

When utilizing the Run:AI model streamer in SGLang to load models directly from a Google Cloud Storage (gs://) bucket, standard text models (e.g., Qwen 32B) start successfully. However, multimodal models (e.g., Kimi K2.5) fail during the server startup phase. The crash occurs when SGLang attempts to initialize the model's multimodal processor.

The Error & When We hit
The SGLang server crashes with a Hugging Face validation error because it passes the raw gs:// URI directly into Hugging Face's configuration loader. The Hugging Face transformers library does not natively support gs:// paths and interprets the URI as a malformed Hugging Face Hub repository ID.

When it happens: This crash happens very early in the model loading process, during the initialization of the TokenizerManager in the _launch_subprocesses phase. Specifically, it fails inside the init_tokenizer_and_processor function before any actual tensor weights are streamed into the GPU.

This results in the following traceback and error:
```
huggingface_hub.errors.HFValidationError: Repo id must be in the form 'repo_name' or 'namespace/repo_name': 'gs://xxxx/models/kimi-k2.5'. Use repo_type argument if needed.
...
OSError: Repo id must be in the form 'repo_name' or 'namespace/repo_name': 'gs://xxxx/models/kimi-k2.5'. Use repo_type argument if needed.
```

Root Cause
The divergence in behavior between text and multimodal models stems from how SGLang initializes them. 

What is a Processor and why is it needed? In the Hugging Face ecosystem, a "processor" groups together multiple preprocessing modules. While a standard text model only needs a tokenizer to convert text strings into token IDs, a multimodal model (like Kimi K2.5) requires an image/audio processor (a feature extractor) to convert raw media into pixel values or feature tensors that the model's vision/audio encoder can understand. SGLang uses this mm_processor to preprocess inputs like image_data and audio_data before passing them to the model.

Why the loading process fails for Multimodal Models:

Text Models (e.g., Qwen 32B): SGLang uses the get_tokenizer function, which contains logic (like download_from_hf) to intercept the cloud URI, download the lightweight configuration files locally, and pass the local path to Hugging Face.
Multimodal Models (e.g., Kimi K2.5): SGLang detects the vision components and routes initialization through [get_processor](https://github.com/sgl-project/sglang/blob/6b2bf66cd9cd0448b0e9f3af8a54e9e10686fdf2/python/sglang/srt/utils/hf_transformers_utils.py#L1226) instead. The [get_processor](https://github.com/sgl-project/sglang/blob/6b2bf66cd9cd0448b0e9f3af8a54e9e10686fdf2/python/sglang/srt/utils/hf_transformers_utils.py#L1226) function lacks the cloud URI interception logic. It directly calls AutoConfig.from_pretrained(tokenizer_name, ...) using the raw gs:// string, leading to the crash.



### Reproduction

Running the following from CLI in VM with 8 GPUs
```
python3 -m sglang.launch_server     --model-path gs://xxxx/models/kimi-k2.5  --load-format runai_streamer --model-loader-extra-config '{"distributed": true}'      --tensor-parallel-size 8     --pipeline-parallel-size 2     --nnodes 2     --node-rank 0     --dist-init-addr xx.xx.xx.xx:YYYY    --trust-remote-code     --enable-p2p-check     --reasoning-parser kimi_k2     --tool-call-parser kimi_k2     --mem-fraction-static 0.85     --chunked-prefill-size 8192     --kv-cache-dtype fp8_e5m2     --enable-flashinfer
```
