# [AMD] Rebalance stage-c-large-8-gpu-mi35x partitions to fix 60-min timeout

URL: https://github.com/sgl-project/sglang/pull/29726
State: closed
Labels: amd, deepseek
Closed at: 2026-07-01T15:17:00Z
Merged at: 2026-07-01T15:17:00Z

## Motivation
`stage-c-test-large-8-gpu-amd-mi35x` (partition 0) has been intermittently
hitting the 60-minute `Run test` step timeout on scheduled `main` runs.
Root cause is partition imbalance, not a code regression. The suite is split
across 2 partitions with an LPT (longest-processing-time) auto-partitioner, but
**every heavy test was registered with the same `est_time=3600`**, so the
partitioner could not balance real runtimes.
The most recently added test, `test_kimi_k25_mxfp4_bcg_mi35x.py` (#27833), is by
far the heaviest (~35 min). With all `est_time`s equal, LPT packed it together
with `test_deepseek_r1_mxfp4_8gpu.py` (and `test_breakable_cuda_graph.py`) into
partition 0, pushing it to ~52 min (up to ~68 min under slow weight-load I/O and
timing out), while partition 1 finished in ~17 min:
| partition | tests | wall time |
| --- | --- | --- |
| 0 | deepseek_r1_mxfp4 + kimi_k25_mxfp4_bcg + breakable_cuda_graph | ~52–68 min |
| 1 | kimi_k25_mxfp4 + qwen3_coder_next | ~17 min |
## Modification
Replace the placeholder `est_time=3600` on the heavy tests with realistic,
measured values so the LPT partitioner balances the two partitions:
| test | est_time (old → new) | measured |
| --- | --- | --- |
| test_kimi_k25_mxfp4_bcg_mi35x | 3600 → 2100 | ~1.9–2.1k s |
| test_deepseek_r1_mxfp4_8gpu | 3600 → 1300 | ~0.86–1.3k s |
| test_kimi_k25_mxfp4 | 3600 → 700 | ~0.58–0.63k s |
| test_qwen3_coder_next_8gpu | 3600 → 450 | ~0.33–0.43k s |
| test_breakable_cuda_graph (amd suite) | 120 → 200 | ~0.18k s |
Resulting deterministic split (in-source `est_time`, no live model file):
| partition | tests | wall time |
| --- | --- | --- |
| 0 | kimi_k25_mxfp4_bcg + breakable_cuda_graph | ~34–40 min |
| 1 | deepseek_r1_mxfp4 + kimi_k25_mxfp4 + qwen3_coder_next | ~29–39 min |
Both partitions now sit well under the existing 60-minute step timeout with
comfortable margin, so no timeout bump is needed.
## Notes
- The partition assignment is deterministic (CI uses in-source `est_time`, not a
  live stats file), so run-to-run execution-time variance does not change the
  split.
- Weight-load I/O variance still moves absolute times, hence the est values are
  set toward the upper end of observed runtimes for margin.
