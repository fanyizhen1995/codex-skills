# [gRPC] proto: add DisaggregatedParams for PD rendezvous

Labels: 
Closed at: 2026-07-13T23:07:53Z
Merged at: 

## Summary

Adds `DisaggregatedParams` to `sglang.runtime.v1` and exposes it as an optional field on `Generate` and `TextGenerate` requests so external routers can populate the bootstrap rendezvous parameters that drive disaggregated prefill/decode.

## What changes

`proto/sglang/runtime/v1/sglang.proto`:

- New `DisaggregatedParams { bootstrap_host, bootstrap_port, bootstrap_room }`
- `optional DisaggregatedParams disaggregated_params = 13` on `TextGenerateRequest`
- `optional DisaggregatedParams disaggregated_params = 12` on `GenerateRequest`

Additions only — no existing field numbers change, no consumer is forced to populate it.

## Why

When a deployment runs in disaggregated prefill/decode, the router needs to pin a single rendezvous identity onto **both** legs so they meet on the KV transfer (the prefill writes its KV under `bootstrap_room` to `(bootstrap_host, bootstrap_port)`; the decode fetches it from the same room). Today there's no way to express that on the gRPC wire — clients have to fall back to the HTTP path or to the legacy SMG-gRPC proto. This adds the missing field on the native proto.

`bootstrap_room` is `int64` so routers have a full 63-bit space for any per-request encoding (e.g. embedding a DP rank in the low bits). Narrower types truncate that.

## Wire-format note

This PR is proto-only. The rust crate scaffold currently on main does not yet consume the new fields — server-side handling will land in a follow-up against the in-flight server stack (#23506–#23509).

This PR is part of a 2-PR stack:

1. **this PR** — `DisaggregatedParams`
2. follow-up — `SubscribeKvEvents` + KV-event messages (separate stream, independent feature)
