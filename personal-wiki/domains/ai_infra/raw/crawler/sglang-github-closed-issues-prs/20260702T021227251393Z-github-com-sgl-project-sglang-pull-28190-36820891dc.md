# fix(precision): do not promote failed runs to the comparison baseline

URL: https://github.com/sgl-project/sglang/pull/28190
State: closed
Labels: high priority, run-ci, bypass-fastfail
Closed at: 2026-07-02T01:19:16Z
Merged at: 2026-07-02T01:19:16Z

Follow-up to #26902 (nightly precision regression test).

## Problem

`_select_latest_run` selects the comparison baseline purely by recency (`push_index`) and ignores `pass_label`. The FAILED branch of the nightly test uploads the regressed tensors to the HF store with `pass_label="failed"` (intended only to preserve the diagnostic diff). Because the next night's `_maybe_hf_fetch` → `fetch_latest_baseline` → `_select_latest_run` picks the most-recent row regardless of label, a failed run becomes the next night's reference baseline.

This **masks persistent regressions after a single night**:

| Night | Event | Result |
|-------|-------|--------|
| N-1 | good baseline | PASS (green) |
| N | regression introduced → today(regressed) vs good baseline | **FAIL** → uploads regressed tensors as `pass_label="failed"` |
| N+1 | baseline = N's failed regressed run → today(still regressed) vs regressed | **PASS** ❌ |
| N+2+ | baseline already regressed | green forever |

CI shows red on night N then green from N+1 onward, making an unfixed regression look self-resolved. This defeats the test's core purpose (continuous regression detection). It manifests in the nominal HF-driven / cross-runner operating mode — i.e. exactly the case the HF store was built for (cold local baseline on a fresh/rotated runner).

This is distinct from the documented "rolling baseline" known-limitation, which covers *gradual* cumulative drift. Here a *sharp* single-night regression — and any persistent regression — is absorbed in one step via failed-run promotion, which also contradicts the FAILED-branch comment's stated intent ("so the diagnostic diff survives on HF").

## Fix

`_select_latest_run` now prefers the most-recent **non-failed** row, falling back to the most-recent failed row only when every candidate failed. Rows missing `pass_label` (legacy) remain usable (`!= "failed"`), preserving backward compatibility.

After the fix:
- Persistent regression: baseline stays pinned to the last good run → the regression is caught *every* night until fixed.
- Once fixed: baseline advances to the newest passed run.

https://github.com/sgl-project/sglang/blob/$(true)/python/sglang/test/precision_baseline_store.py

## Testing

Added 4 unit tests to `TestSelectLatestRun` (`test/registered/unit/test_precision_baseline_store.py`):
- `test_prefers_older_passed_over_newer_failed`
- `test_prefers_baseline_established_over_newer_failed`
- `test_falls_back_to_failed_when_only_failed`
- `test_missing_pass_label_treated_as_usable` (legacy backward compat)

All existing `TestSelectLatestRun` cases still pass (no behavior change when no `failed` rows are present). Verified `_select_latest_run` against the full existing + new assertion set plus end-to-end masking scenarios (persistent-regression pinned to last good; baseline advances after fix).
