# [Intel XPU] Initially add nightly GSM8K accuracy tests for Llama-3.1-8B (TP=2) and Qwen3-32B (TP=4)

URL: https://github.com/sgl-project/sglang/pull/28908
State: closed
Labels: run-ci, run-ci-extra
Closed at: 2026-07-01T08:27:25Z
Merged at: 2026-07-01T08:27:25Z

Wire up the Intel XPU nightly flow on `intel-bmg-nightly`. Every suite/job is registered via `register_xpu_ci(... nightly=True)` and dispatched by `test/run_suite.py`, where the digit in the suite name = TP size, not host card count.

## Suites and tests

| Suite | TP | Test | Threshold |
|---|---|---|---|
| `nightly-xpu-2-gpu` | 2 | `test_xpu_llama_3_1_8b.py` | GSM8K ≥ 0.80 |
| `nightly-xpu-4-gpu` | 4 | `test_xpu_qwen3_32b.py` | GSM8K ≥ 0.85 |

Both jobs serialize on the single `intel-bmg-nightly` host. A `check-all-jobs` aggregator fails the workflow if any nightly suite job failed or was cancelled.

## Plumbing

- `python/sglang/test/xpu/simple_eval_gsm8k_xpu_mixin.py` launches the SGLang server with `--device xpu --attention-backend intel_xpu`, runs `simple_eval_gsm8k.GSM8KEval` (same evaluator as the AMD/CUDA nightlies), asserts `accuracy >= threshold`, and renders a Markdown table to `$GITHUB_STEP_SUMMARY`.
- `python/sglang/test/xpu/test_xpu_utils.py` writes the per-run summary table.
- New input `continue_on_error` (default `true`) on `workflow_dispatch` / `workflow_call` so a single flaky model doesn't fail the whole pipeline; the schedule trigger always passes `--continue-on-error` to `run_suite.py`.

## Test plan

- [ ] `workflow_dispatch` against this branch — verify both jobs spin up on `intel-bmg-nightly` and `check-all-jobs` reflects each result.
- [ ] Confirm `python3 run_suite.py --hw xpu --suite nightly-xpu-{2,4}-gpu --nightly` discovers exactly the registered tests for each suite.
- [ ] Spot-check the rendered `$GITHUB_STEP_SUMMARY` table after one full run; tune accuracy thresholds in a follow-up if any model is consistently below baseline.
