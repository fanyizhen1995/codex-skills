# [dLLM] Make FDFO a framework capability for all dLLM algorithms

URL: https://github.com/sgl-project/sglang/pull/27551
State: closed
Labels: documentation, high priority, deepseek, run-ci, diffusion, bypass-fastfail
Closed at: 2026-07-11T03:05:06Z
Merged at: 2026-07-11T03:05:05Z

## Motivation

This PR supersedes #25280. It generalizes **FDFO (First-Done-First-Out)** scheduling for diffusion LLM (dLLM) inference from a single bespoke algorithm into a **framework-level execution mode that any dLLM algorithm can opt into**, including algorithms that carry cross-step state.

### Problem

In synchronous dLLM batching, all requests in a batch advance in lockstep. High-confidence requests that converge in few denoise steps are forced to wait for low-confidence, long-tail requests before they can leave the batch. This head-of-line blocking leaves the GPU idle and caps throughput. FDFO frees a request the moment its decode block is fully resolved, without waiting for slower requests in the same batch.

### Why a framework capability

#25280 bundled FDFO into a dedicated `LowConfidenceFDFO` algorithm class, which conflated two orthogonal concerns and duplicated the token-selection logic:

- **token-selection strategy** — which masked positions to unmask this denoise step (low-confidence threshold, joint-threshold, …);
- **execution mode** — denoise a whole block synchronously vs. yield to the scheduler after each step (FDFO).

This PR separates them so FDFO works for every algorithm, and adds the per-request state plumbing needed for **stateful** algorithms.

## Design

### Strategy vs. execution split

`DllmAlgorithm` now defines a single-step strategy; the base class owns both execution loops:

- `step(forward_batch, full_logits, states) -> done_flags` — one denoise step; advances `input_ids`/`states` in place and returns, per block, whether it was already complete on entry (so this forward persisted its final KV cache).
- `init_step_state(forward_batch)` / `max_steps(block_size)` — per-block cross-step state and the step budget.
- `run(...)` (base) branches on the mode: synchronous loops `forward → step` until the block resolves; FDFO does one `forward → step` and returns control to the scheduler.

`LowConfidence` (stateless) and `JointThreshold` (stateful) are reduced to `step()`; their internal loops are removed. `LowConfidenceFDFO` is deleted.

### Orthogonal selection

A new `--dllm-fdfo` flag drives `DllmConfig.first_done_first_out_mode`, independent of `--dllm-algorithm`. Any algorithm runs in either mode.

### Cross-step state carrier (stateful algorithms)

- `Req.dllm_incomplete_ids` — the in-progress block tokens (universal).
- `Req.dllm_algo_state` — opaque algorithm-private cross-step state. Stateless algorithms keep it `None`; `JointThreshold` stores `{post_edit_steps, prompt_mask}`. The framework carries it across FDFO rounds via `GenerationBatchResult.dllm_algo_state` and clears it on block completion.

`process_batch_result_dllm_fdfo` manages the per-round lifecycle and releases the masked KV slots of carried blocks.

## Correctness

At **batch size 1**, FDFO and synchronous execution run identical forward shapes, so a correct implementation must produce **byte-identical** output. Verified on `inclusionAI/LLaDA2.0-mini` (H200), temperature 0:

- `LowConfidence`: FDFO == sync, **identical** on all probe prompts (multi-block).
- `JointThreshold` (stateful): FDFO == sync, **identical** on all probe prompts — confirming the cross-step state carry is correct.

At batch > 1, scores fluctuate by ±0.01 between runs for **both** sync and FDFO (kernel batch non-invariance, incl. MoE routing); the two distributions overlap, so this is run-to-run noise rather than an FDFO regression.

## Performance (H200, TP=1, LLaDA2.0-mini, GSM8K, 200 examples)

LowConfidence, output throughput:

| max-running-requests | sync | FDFO | speedup |
|---|---|---|---|
| 4  | 525 tok/s  | 680 tok/s  | **1.30×** |
| 16 | 918–959 tok/s | 1276–1378 tok/s | **~1.45×** |

Scores stay in the same band across modes (sync 0.905–0.92, FDFO 0.90–0.915).
