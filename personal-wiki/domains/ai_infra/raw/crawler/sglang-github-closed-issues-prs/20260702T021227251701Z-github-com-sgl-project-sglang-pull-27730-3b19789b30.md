# [AMD]: docker(rocm) bump Mooncake to latest main + enable multi-protocol

## Motivation

The old Mooncake pin lacks the multi-protocol/hip_transport machinery: with `USE_HIP=ON` the engine became XGMI-only and broke RDMA with `NotSupportedTransport`. Latest main plus `ENABLE_MULTI_PROTOCOL` lets RDMA (cross-node) and HIP/XGMI (intra-node) coexist with runtime transport selection. A commit bump alone is insufficient — the cmake flag is required.

## Modifications

Upgrade `MOONCAKE_COMMIT` from v0.3.7.post2 (`b6a841dc`, 2025-11-04) to latest main (`d8f35569`, 2026-06-09), and build with `-DENABLE_MULTI_PROTOCOL=ON -DWITH_STORE=ON` in `docker/rocm.Dockerfile`.

### Runtime transport selection (XGMI <-> RDMA)

Selection is **automatic per peer** — there is no rebuild, restart, or manual flag to switch transports. With the multi-protocol build, every transfer-engine instance publishes *both* protocols (`rdma` and `hip`) on its segment. For each transfer, `MultiTransport::mp_selectTransport` matches the initiator's preferred protocol against the protocols the target segment advertises and routes accordingly:

- **Intra-node** (prefill and decode GPUs on the same host) → HIP/XGMI
- **Cross-node** (prefill and decode on different hosts) → RDMA (e.g. bnxt_re / mlx5)

Example: a single multi-node PD-disaggregated deployment using one image. No per-transport config differences — the same launch command works everywhere:

```bash
# Prefill (any node)
python3 -m sglang.launch_server --model-path <model> \
  --disaggregation-mode prefill \
  --disaggregation-transfer-backend mooncake \
  --disaggregation-ib-device bnxt_re0,bnxt_re1,...  # RDMA NICs for cross-node

# Decode (any node)
python3 -m sglang.launch_server --model-path <model> \
  --disaggregation-mode decode \
  --disaggregation-transfer-backend mooncake \
  --disaggregation-ib-device bnxt_re0,bnxt_re1,...
```

At runtime, a transfer to a decode peer on the *same* host is carried over XGMI, while a transfer to a peer on a *different* host is carried over RDMA — both from the same process, chosen automatically. Before this change, a HIP build advertised only the HIP protocol, so cross-node peers failed with `NotSupportedTransport`.

Verify which transport is active from the engine logs, e.g. `Topology discovery complete. Found N HCAs` confirms RDMA is registered, and the absence of `NotSupportedTransport` confirms peers negotiated a common transport.
