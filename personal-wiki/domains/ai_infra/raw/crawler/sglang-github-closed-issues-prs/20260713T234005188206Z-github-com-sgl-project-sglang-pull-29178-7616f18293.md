# [AMD] Honor load_full_w2 for grouped w2 scales (fixes Kimi-K2.6 INT4 MoE load on ROCm)

URL: https://github.com/sgl-project/sglang/pull/29178
State: closed
Labels: run-ci
Closed at: 2026-07-13T09:58:11Z
Merged at: 

## Motivation

`nightly-8-gpu-kimi-k26` and `nightly-8-gpu-mi35x-kimi-k26` fail loading `moonshotai/Kimi-K2.6` (compressed-tensors INT4 group-quant MoE, `group_size=32`) at TP=8:

```
File "python/sglang/srt/layers/moe/fused_moe_triton/layer.py", line 566, in _load_w2
    loaded_weight = loaded_weight.narrow(shard_dim, shard_size * tp_rank, shard_size)
IndexError: start out of range (expected to be in range of [-64, 64], but got 320)
```

**Root cause:** On ROCm, Kimi-K2.6's INT4 group MoE uses `CompressedTensorsWNA16TritonMoE`. That scheme keeps the **full, unsharded** `w2_weight_scale` on every TP rank and marks the param with `load_full_w2=True` (group axis = `intermediate_size * moe_tp_size / group_size = 2048 / 32 = 64`).

The generic `FusedMoE` weight loader **ignores that flag**: `_weight_loader_impl` → `_load_model_weight_or_group_weight_scale` → `_load_w2` always narrows the scale by `shard_size * tp_rank`. For the full (unsharded) scale this offset overshoots the 64-element group axis — at TP=8 the offset reaches `64 * 5 = 320`, raising the `IndexError` and crashing model load.

## Modifications

Thread the existing `load_full_w2` param attribute through the loader so the full scale is copied whole instead of narrowed:

- `_weight_loader_impl` (GROUP/BLOCK scale branch): pass `load_full_w2=getattr(param, "load_full_w2", False)`.
- `_load_model_weight_or_group_weight_scale`: accept `load_full_w2` and forward it to `_load_w2`.
- `_load_w2`: when `load_full` is set, `expert_data.copy_(loaded_weight)` and return early (skip the TP narrow).

Only the `load_full_w2` path is affected; every other MoE quant scheme is unchanged. Net diff: +12 lines, one file.

## Accuracy Tests

Not applicable to model outputs directly — this fixes weight **loading** so the model can start at all (it currently crashes at load). The copied scale is the full, correct tensor the scheme expects, identical to the non-TP-sharded layout it allocates.

**Offline-verified on AMD MI300X (gfx942)** by driving `_load_w2` with the Kimi-K2.6 w2-scale dims (full group axis = 64, `tp_rank=5`):

```
--- stock behavior (narrow by shard_size * tp_rank) ---
  IndexError: start out of range (expected to be in range of [-64, 64], but got 320)   # exact nightly error reproduced
--- with this fix (load_full=True) ---
  copied whole scale, equal=True
  VERDICT: PASS
```

## Speed Tests and Profiling

No performance impact: the change only selects copy-vs-narrow at weight-load time for the full-scale case; no inference/runtime path changes.
