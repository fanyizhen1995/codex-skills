# [MLX] Fix single-token chunked-prefill continuation misrouted as decode

URL: https://github.com/sgl-project/sglang/pull/30181
State: closed
Labels: apple-silicon
Closed at: 2026-07-07T03:31:33Z
Merged at: 2026-07-07T03:31:33Z

## Motivation

On the MLX (Apple Silicon) backend, a chunked-prefill **continuation whose final chunk is exactly one token** is silently misrouted to the decode path, corrupting the request's output.

When a prompt's length is `k * chunked_prefill_size + 1`, the final continuation chunk carries exactly one token. `MlxTpModelWorker` routing keyed on `seq_len > 1` to tell a prefill continuation apart from a genuine single-token decode step mixed into the batch — but a 1-token continuation is indistinguishable from a decode by length alone, so it falls into the decode branch.

The decode path does not consume the batch's input token; it feeds the model the token stored from the previous chunk, which is the model's own argmax prediction for that position (the extend path pops this stale token, decode does not). As a result:

- the real last prompt token is dropped and never enters the model;
- generation continues conditioned on a corrupted prompt;
- with radix cache enabled, the wrong-token KV is committed under the correct prefix and can poison prefix reuse for other requests.

The corruption is masked whenever the model's prediction happens to equal the real token (common for natural text), so it only surfaces when the prompt ends on an unexpected token — which is why it has gone unnoticed.

### Repro (mechanistic)

Model `mlx-community/Qwen2.5-0.5B-Instruct-4bit`, prompt `"The quick brown fox jumps over the lazy cat"` (9 tokens; after "…the lazy" the model predicts `" dog"` = 5562, but the real last token is `" cat"` = 8251). Instrumenting the routing branch:

- `--chunked-prefill-size 8` (final chunk = 1 token): the continuation is routed to **decode**, the real token `8251` is discarded, and the model is fed its stored guess `5562`.
- `--chunked-prefill-size 7` (final chunk = 2 tokens): correctly routed to **extend**, real tokens `[15678, 8251]` passed in.

A plain output diff (chunked vs. unchunked) is not a reliable check here: on quantized models, greedy decoding diverges from ordinary floating-point/kernel noise even without the bug. The routing decision itself is the deterministic signal.

## Modifications

Route by request state instead of chunk length, at both routing sites:

- Compute `decoding_rids = {r.rid for r in (batch.decoding_reqs or [])}` — the genuine decode requests the scheduler mixes in via mixed-style chunked prefill.
- A request we have already seen is a decode step iff its rid is in `decoding_rids`; otherwise it is a prefill continuation and takes the extend path regardless of length.
- Applied to both `_forward_batch_generation_mlx` (sync) and `_async_extend_batch` (async).

Behavior-preserving for every case except the 1-token continuation: genuine mixed decodes (always `extend_len == 1`) stay on the decode path, and multi-token continuations are unchanged.

## Accuracy Tests

- Added `test/registered/unit/hardware_backend/mlx/test_tp_worker_routing.py`: a mock-runner unit test (no model load) pinning the routing contract — 1-token continuation → extend, multi-token continuation → extend, genuine mixed decode → decode. Verified to fail on the pre-fix routing and pass after the fix.
- Re-confirmed end-to-end with the real scheduler (see repro above): the 1-token final chunk now routes to extend and passes the real token.

## Speed Tests and Profiling

No measurable impact: the only added work is building a small set of request ids once per extend batch (not per token).
