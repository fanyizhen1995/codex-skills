# Regression: DeepGEMM 0.1.4 UE8M0 scale assert loading GLM-5.2-FP8 with fp8_gemm_runner_backend=auto

### Summary

After the `sgl-deep-gemm` bump in #29554 (`0.1.3` -> `0.1.4`), serving `zai-org/GLM-5.2-FP8` on Blackwell with the default FP8 GEMM backend selection can fail during model load with a DeepGEMM UE8M0 scale assertion.

The same run worked a few days earlier before the environment picked up `sgl-deep-gemm==0.1.4`. Downgrading only `sgl-deep-gemm` back to `0.1.3` removes the failing assert from the installed DeepGEMM headers and is a temporary workaround.

### Model / launch shape

Model:

```text
zai-org/GLM-5.2-FP8
```

Relevant server args from the failed run:

```text
--tp 8
--quantization fp8
--kv-cache-dtype fp8_e4m3
--attention-backend dsa
--dsa-prefill-backend trtllm
--dsa-decode-backend trtllm
--moe-runner-backend flashinfer_trtllm
--fp8-gemm-runner-backend auto
```

Hardware/runtime context:

```text
Blackwell / SM100-class GPU
CUDA 13 stack
sgl-deep-gemm==0.1.4
```

### Observed behavior

During startup, SGLang logs:

```text
DeepGemm is enabled but the scale_fmt of checkpoint is not ue8m0. This might cause accuracy degradation on Blackwell.
```

After all model shards finish loading, DeepGEMM fails with repeated device-side assertions:

```text
Assertion failed: /usr/local/lib/python3.12/dist-packages/deep_gemm/include/deep_gemm/impls/smxx_layout.cuh:131,
condition: (values[j] & 0x807fffffu) == 0
```

The later Python stack reports an async CUDA launch failure around post-load processing, e.g.:

```text
torch.AcceleratorError: CUDA error: unspecified launch failure
```

The full log is no longer available, but the crash happened during model load/postprocess, before serving traffic. It was not the PP-parallel DeepGEMM compile warmup path.

### Why #29554 does not appear to cover this case

#29554 changed DeepGEMM warmup dummy scale tensors from `torch.empty(...)` to `torch.ones(...)`, which fixes uninitialized dummy scales in `pp_parallel_deep_gemm_warmup`.

This failure is different:

- `pp_size=1`, so the PP-parallel DeepGEMM warmup path should not run.
- The checkpoint has FP8 block scales but does not declare `scale_fmt: ue8m0`.
- `fp8_gemm_runner_backend=auto` still selects DeepGEMM on Blackwell.
- The assert is from DeepGEMM requiring FP32 scale factors to be bit-exact UE8M0-compatible: zero sign and mantissa bits.

### Likely problematic area

SGLang has a checkpoint-scale requant path for DeepGEMM:

```python
requant_block_scale_ue8m0_for_deepgemm(...)
```

which calls:

```python
requant_weight_ue8m0(...)
```

The local copied helper currently computes UE8M0-ish scales using floating math:

```python
def ceil_to_ue8m0(x: torch.Tensor):
    return torch.pow(2.0, torch.ceil(torch.log2(x.abs())))
```

DeepGEMM 0.1.4 appears stricter and requires bit-exact exponent-only FP32 values. DeepGEMM's own helper uses bit manipulation instead, roughly:

```python
def ceil_to_ue8m0(x: torch.Tensor):
    bits = x.abs().float().view(torch.int32)
    exp = ((bits >> 23) & 0xFF) + ((bits & 0x7FFFFF) != 0).to(torch.int32)
    return (exp.clamp(1, 254) << 23).view(torch.float32)
```

### Workarounds

Either of these avoids the issue:

```bash
--fp8-gemm-runner-backend flashinfer_trtllm
```

or:

```bash
SGLANG_ENABLE_JIT_DEEPGEMM=0
```

Downgrading also avoids it:

```bash
pip install --force-reinstall --no-deps sgl-deep-gemm==0.1.3
```

### Possible fixes

One of:

1. Make SGLang's `ceil_to_ue8m0` bit-exact, matching DeepGEMM's implementation, so checkpoint-scale requantization produces valid UE8M0 scale bits for DeepGEMM 0.1.4.
2. In `fp8_gemm_runner_backend=auto`, avoid choosing DeepGEMM for FP8 checkpoints where `quantization_config.scale_fmt != "ue8m0"` unless the requant path is known to be safe.
3. Turn the current warning into a backend fallback on Blackwell when auto-selecting FP8 GEMM.

The main concern is that the current warning says accuracy may degrade, but with `sgl-deep-gemm==0.1.4` this can become a hard startup crash.
