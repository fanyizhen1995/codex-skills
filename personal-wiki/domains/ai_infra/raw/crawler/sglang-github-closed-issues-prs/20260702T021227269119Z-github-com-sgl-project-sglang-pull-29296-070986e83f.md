## Motivation

<!-- Describe the purpose and goals of this pull request. -->
Encoder-only server health checks can hang when `--enable-mm-global-cache` is used with `tp-size > 1`.

The root cause is that rank 0 handles `/health` by directly calling `encode()`, while non-zero TP ranks receive the dummy request from the scheduler socket and previously dispatch it through the regular `encode_request()` path. For global cache, that regular path may enter cache coordination logic that rank 0 does not join, causing the encode server health check to hang.

This can also cause the language-only server's encoder bootstrap health checker to evict the encoder URL after consecutive failures. For example:

```text
[2026-06-25 17:00:14] Dispatching 1 mm items to 1 encoder(s) ['http://127.0.0.1:30000'] for request 4453b43bf6824d34ac82cd5549ae7ee2
[2026-06-25 17:00:14] Waiting for input self.embedding_port = 32679
[2026-06-25 17:00:14] self.num_items_assigned = OrderedDict({<Modality.IMAGE: 1>: [1]})
[2026-06-25 17:00:14] Preparing to send to http://127.0.0.1:30000/scheduler_receive_url with part_req_id=4453b43bf6824d34ac82cd5549ae7ee2_local_part_0
[2026-06-25 17:00:14] Concurrently sending 1 requests...
[2026-06-25 17:00:20] Health check evicted 1 encoder(s) after 3 consecutive failures: ['http://127.0.0.1:30000']
```
After that, the encode server can remain stuck and the language server no longer has an available encoder URL.

## Modifications

<!-- Detail the changes made in this pull request. -->
- Route encoder health-check dummy requests on non-zero TP ranks through `encoder.encode()` by reusing the existing `HEALTH_CHECK_RID_PREFIX`.
- Keep regular encoder worker requests on the existing `encode_request()` path.
