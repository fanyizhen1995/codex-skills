# Feat/flexkv main connector

URL: https://github.com/sgl-project/sglang/pull/29701
State: closed

Closed at: 2026-07-07T07:35:53Z
Merged at: 2026-07-07T07:35:53Z

## Motivation
  
  The scheduler ticks `tree_cache.check_hicache_events()` only when `enable_hierarchical_cache` is True. `FlexKVRadixCache` (added by `--enable-flexkv`) implements the same hook to drain its async fire-and-forget stores (`_drain_completed_stores()` → `dec_lock_ref`), but `--enable-flexkv` does not flip the hierarchical-cache flag, so the hook is never called.

  The consequence under load: every `cache_finished_req()` does `inc_lock_ref(new_last_node)` to protect the source slots while FlexKV stores them. Without the periodic drain, those locks are never released → `evictable_size` collapses to 0 → the prefill adder returns `NO_TOKEN` for every queued request → the scheduler idles in `recv_pyobj` while the client hangs on "Waiting for N tasks to finish".

  Reproduced with `mem_fraction_static=0.3`, 40 concurrent users from `LMBenchmark/synthetic-multi-round-qa`. `py-spy dump` on the scheduler process shows it bouncing between `recv_requests()` (empty) and `_abort_on_waiting_timeout` with `batch=None`; the FlexKV server log shows D2H transfers stop completely once all running reqs finish and the allocator wedges.

  ## Modifications

  `python/sglang/srt/managers/scheduler.py`:
  
  1. Cache `self.enable_flexkv = getattr(server_args, "enable_flexkv", False)` next to the other cache-related flags in `Scheduler.__init__`.
  2. Gate `check_hicache_events()` on `enable_hierarchical_cache or enable_flexkv`. This is the load-bearing change.
  3. Apply the same OR to the softer `batch_is_full` handling on `NO_TOKEN`, so the FlexKV path keeps trying to admit waiters as inflight stores complete (matching the hicache behavior the design copies from).

  `FlexKVRadixCache` itself is unchanged — its `check_hicache_events()` already does the right thing, it just was never called.

  ## Validation

  `bench/LMBenchmark/synthetic-multi-round-qa`, Qwen3-8B on H20, `mem_fraction_static=0.3`, `max_running_requests=16`, 40 users × 6 rounds, 90s bench, target qps=2 (server-bound, achieved ~0.7–0.9 reqs/s for all configs):

  | Config | TTFT | Output tok/s | Gen tok/req/s | Processing | H2D / D2H |
  | --- | --- | --- | --- | --- | --- |
  | baseline | 49.6s | 67.7 | 131.1 | 0.68 reqs/s | — |
  | `--enable-hierarchical-cache` | 39.4s | 88.7 | 131.2 | 0.89 reqs/s | — |
  | `--enable-flexkv` (before patch) | hang | — | — | — | 0 / 26 |
  | `--enable-flexkv` (after patch) | **39.4s** | **88.6** | **130.8** | **0.89 reqs/s** | **139 / 176** |

  Post-patch, FlexKV matches HiCache on every metric within noise; both deliver ~26% higher throughput than baseline. The 139 H2D transfers confirm the connector is actually serving requests from the CPU pool — before the patch it was 0 because the connector wedged before any request hit round 2.

  A lighter run (`mem_fraction_static=0.4`, 16 users, qps=2, 90s) was also used during diagnosis: 8.94s TTFT, 197.7 output tok/s, 92.2 tok/req/s gen, 1.98 reqs/s — confirming clean behavior under non-thrashing load.

## Benchmark: SWE-bench Lite (oracle)                                                     
                                                                                            
  Reproducible three-way comparison — same server binary, same prompt set, three configs (`baseline` / `--enable-hierarchical-cache` / `--enable-flexkv`). Harness lives at           
  [`python/sglang/srt/mem_cache/storage/flexkv/benchmarks/`](python/sglang/srt/mem_cache/storage/flexkv/benchmarks/); reproduce with `bash run_swebench.sh` after `export
  MODEL=/path/to/Qwen3-8B`.                                                                                                                                                           
                                              
  **Setup:** Qwen3-8B on 1× H20. `--attention-backend triton --mem-fraction-static 0.32 --max-running-requests 32 --chunked-prefill-size 16384 --context-length 32000`. HiCache /
  FlexKV each get 80 GB of CPU pool.

  **Workload:** 120 prompts sampled from [`princeton-nlp/SWE-bench_Lite_oracle`](https://huggingface.co/datasets/princeton-nlp/SWE-bench_Lite_oracle), input length ≤ 28 k tokens (p50
   = 7088, max = 27961). Two passes — pass 1 populates the host cache, pass 2 is the measured run. `qps=2.0`, `concurrency=24`, `max_new_tokens=32`, `temperature=0`.

  ### Warm-pass results

  | Config | TTFT avg / p50 / p90 / p99 | E2E p50 | Throughput | Output tok/s | H2D / D2H |
  | --- | --- | --- | --- | --- | --- |
  | baseline                      | 6.86 / 8.04 / 9.88 / 10.89 s | 8.15 s | 1.86 req/s | 37.7 | — |
  | `--enable-hierarchical-cache` | **0.04 / 0.04 / 0.06 / 0.06 s** | 0.23 s | 2.02 req/s | 40.8 | — |
  | `--enable-flexkv`             | **0.05 / 0.05 / 0.07 / 0.08 s** | 0.24 s | 2.02 req/s | 40.8 | 86 / 155 |

  TTFT p50 drops **~170×** vs baseline; throughput improves **+8.6 %**. FlexKV and HiCache are within noise of each other.

  ### Server-side hit-rate sanity check

  Parsing `ReqTimeStats` in the sglang log across all warm-pass requests with non-empty output:

  | Config | warm reqs | `cached_input_len ≈ input_len` (>90 %) | Mean hit rate |
  | --- | --- | --- | --- |
  | baseline                      | 76 | 0 / 76   | 1.4 % (system-prompt header only) |
  | `--enable-hierarchical-cache` | 76 | **76 / 76** | **100.0 %** |
  | `--enable-flexkv`             | 76 | **76 / 76** | **100.0 %** |

  FlexKV additionally emits 86 `H2D transfer` log lines totaling ~60 GB of KV pulled back from the CPU pool — confirming loadbacks actually fired end-to-end.

  ### Output correctness

  Byte-level diff of generated text over a 32-prompt subset, `temperature=0`:

  | Comparison | identical | diverge |
  | --- | --- | --- |
  | baseline cold vs baseline warm | 32 / 32 | 0 (fully deterministic without cache) |
  | hicache warm vs baseline warm  | 29 / 32 | 3 |
  | flexkv  warm vs baseline warm  | 29 / 32 | 3 (mostly the same 3 as hicache) |

  The ~10 % divergence at `temperature=0` is a known KV-cache-reuse artifact caused by floating-point non-associativity between "prefill in place" and "load pre-computed KV" — it
  affects the mainline `--enable-hierarchical-cache` at the same rate and is not FlexKV-specific.
