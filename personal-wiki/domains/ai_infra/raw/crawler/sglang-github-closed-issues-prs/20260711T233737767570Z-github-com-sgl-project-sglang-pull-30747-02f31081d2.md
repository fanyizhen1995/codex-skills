# Fix: add grammar sync in PP for structured output

## Motivation

<!-- Describe the purpose and goals of this pull request. -->

Fix PP scheduler divergence when structured-output requests (requests using `response_format`) cause grammar-ready inconsistency across PP schedulers.

In PP mode, each worker has its own scheduler. Grammar compilation runs asynchronously on CPU, so the same request's grammar can become ready at different wall-clock times on different PP ranks.
However, `GrammarManager.get_ready_grammar_requests()` only synchronized ready/failed grammar state within TP/DP groups, then immediately popped requests from `grammar_queue`.

**This could make PP ranks schedule different batches, which in turn caused sglang to crash.**

This PR adds a synchronization across PP ranks for grammar ready/failed decisions to prevent PP scheduler divergence.

Fixes #28424


## Modifications

<!-- Detail the changes made in this pull request. -->

**Grammar Manager**
- Add PP synchronization (similar to `_pp_sync` in #27285) for grammar ready/failed decisions so only PP0 polls grammar futures, performs TP synchronization, and propagates the final ready/failed indexes to later PP ranks. Later PP ranks wait for grammar completions on the result from PP0.
- Add unit test for grammar PP sync behavior.

**PP Distributed Communication**
- Add optional tag to `GroupCoordinator.send_object` and `recv_object`. The default value (tag=0) preserves the original default behavior. (`parallel_state.py`)
- Add PP P2P tags for _pp_sync_ outside schduler-loop, to avoid mixing with scheduler PP communication. (`communication_tags.py`, new file)
- Replace hardcoded PP sync tag values in hiradix/unified_radix with the above p2p tag.

## Speed Tests and Profiling

<!-- If this pull request impacts inference speed, provide benchmarking and profiling results. -->

Test with sglang.bench_serving (also with specifying response_format).
No impact on normal cases.

Note: In extreme cases (high-concurrency of requests with structured output), performance may degrade. E.g., for Qwen3-8B, PP 4, 128 concurrency, and disable grammar cache, less than 10% degradation compared to normal requests.
In this case, many grammar tasks are added to `ThreadPoolExecutor`, and different schedulers may execute them in different orders.
By the way, similar performance degradation is also observed in pure TP.
