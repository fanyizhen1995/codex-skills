# [Platform] Route pin memory availability through current_platform

## Motivation

Out-of-tree platforms can currently implement `SRTPlatform.is_pin_memory_available()`, but the common `is_pin_memory_available()` helper returns `False` before consulting the platform whenever `torch.cuda.is_available()` is false.

This prevents non-CUDA OOT hardware plugins from explicitly opting into pinned memory support through the platform interface.

## Modifications

- Route OOT platform pin-memory availability through `current_platform.is_pin_memory_available()` only when the OOT platform overrides the base method.
- Preserve the historical fallback behavior for OOT platforms that do not override the hook: they still use the existing `torch.cuda.is_available()` check.
- Keep `device="cpu"` returning `False`.
- Add unit tests for:
  - OOT override returning `True`
  - OOT override returning `False`
  - OOT without override preserving CUDA available/unavailable behavior
  - CPU device handling
  - in-tree CUDA available/unavailable behavior
  - base `SRTPlatform` default compatibility
