# [HiCache][AMD] Add UMBP tiered DRAM + SSD L3 storage backend with hugepage host allocator  

URL: https://github.com/sgl-project/sglang/pull/25377
State: closed
Labels: amd, run-ci, run-ci-extra
Closed at: 2026-07-01T14:21:38Z
Merged at: 2026-07-01T14:21:38Z

## Summary

Adds a new HiCache storage backend, `mori`, that offloads evicted KV pages to a host-side L3 tier managed by the [`mori`](https://github.com/ROCm/mori) library's `umbp` host-memory pool subsystem. (`mori` is AMD's EP / IO backend for MoE serving on MI3xx; `umbp` is its tiered host-memory pool component.)

This PR includes:

- `UMBPStore` as a zero-copy HiCache storage backend.
- A hugepage-capable host allocator backed by mori's UMBP host-memory pool.
- Integration with HiCache storage backend selection via `--hicache-storage-backend mori`.
- DRAM tier support, plus SSD code-path support.
- Focused Python unit tests for the allocator and store behavior.

This PR intentionally does not include:

- Monitoring / Grafana dashboard changes.
- Dummy or noop forward stress-test mode.
- Broader HiCache eviction robustness fixes that are not specific to the mori backend.
- Environment-specific benchmark launch scripts beyond the generic validation notes below.

## Usage

Enable the backend with:

```bash
--enable-hierarchical-cache
--hicache-storage-backend mori
--hicache-storage-backend-extra-config @mori.json
--hicache-write-policy write_back
--hicache-mem-layout page_first
```

(both `@mori.json` and the inline `'{...}'` forms are accepted by `--hicache-storage-backend-extra-config`.)

Example `mori.json`:

```json
{
  "dram_capacity_bytes": 17179869184,
  "ssd_enabled": false,
  "eviction_policy": "prefix_aware_lru",
  "auto_promote_on_read": true,
  "master_address": "127.0.0.1:7777",
  "node_address": "127.0.0.1"
}
```

For distributed UMBP mode, start `umbp_master` before launching SGLang:

```bash
nohup umbp_master 0.0.0.0:7777 > umbp_master.log 2>&1 &
```

## Motivation

On MI355X, HBM capacity is the bottleneck for long-context multi-DP serving. HiCache's existing HBM tier cannot retain enough reused-prefix KV for realistic agentic and multi-turn workloads, so a large fraction of cacheable prefill work is recomputed.

The mori backend adds a larger host-side L3 tier on top of HiCache's existing HBM L2. In the validation below, each rank uses a DRAM-backed L3 tier, with an optional SSD path available for spill / smoke testing.

## Implementation Notes

- `UMBPStore` follows the HiCache zero-copy v1 interface.
- `page_first` layout is required for the current UMBP put/get path.
- `master_address` enables distributed UMBP client mode and connects ranks to the `umbp_master` process.
- `dram_page_size` can be provided as an operator override; otherwise the store derives it from the host KV page layout when possible.
- SSD behavior is controlled through fields such as `ssd_enabled`, `ssd_storage_dir`, `ssd_capacity_bytes`, and `ssd_backend`.

## Validation

### Correctness

DeepSeek-R1-0528 MXFP4, gsm8k, single node with 8x MI355X, `lm-eval==0.4.4`, `--num_fewshot 5`, page size 64, limit 200, concurrency 64.

| Run | flex-extract | stderr | strict-match | stderr |
| --- | ---: | ---: | ---: | ---: |
| Baseline, no mori | 0.960 | +/- 0.0139 | 0.960 | +/- 0.0139 |
| mori `write_back` | 0.980 | +/- 0.0099 | 0.970 | +/- 0.0121 |

mori matches the no-mori baseline within stderr: delta = +0.020 on flex-extract, combined stderr ~= 0.017 (~1.2 sigma).

### Cache Hit / TTFT

DeepSeek-V3-5layer synthetic multi-turn prefix-reuse workload, single node with 8x MI355X. The workload uses `bench_multiturn.py` with 20 rounds, 64 clients, 32 max parallel requests, 2048-token request length, 430-token sub-question length, and 1-token output.

| Case | HiCache tiers | cache hit rate | avg TTFT | p90 TTFT |
| --- | --- | ---: | ---: | ---: |
| 1 | HBM only, no storage backend | 1.08% | 175.6 ms | 510.2 ms |
| 2 | HBM + mori DRAM | 48.68% | 50.0 ms | 89.2 ms |
| 3 | HBM + mori DRAM + SSD config smoke | 47.11% | 52.7 ms | 96.5 ms |

Headline result for case 1 to case 2: enabling the mori DRAM tier improves cache hit rate by +47.6 percentage points and reduces average TTFT by about 3.5x.

Case 3 keeps `ssd_capacity_bytes` at `0` and `ssd_enabled` at `false`, so it validates the SSD-related configuration path without provisioning an active SSD tier.

> Sizing note: the correctness setup uses `--hicache-ratio 1.0` (HBM L2 sized to fit the working set, so L2 is not the bottleneck and the test isolates UMBP put/get correctness). The cache-hit setup uses `--hicache-size 2` (2 GB/rank fixed L2) to force L2->L3 spill quickly so the L3 contribution is observable.

### Unit Tests

These tests do not require GPU or a running `umbp_master`; mori dependencies are mocked:

```bash
pytest python/sglang/srt/mem_cache/storage/umbp/test_umbp_host_allocator.py \
       python/sglang/srt/mem_cache/storage/umbp/test_umbp_store.py
```

- `test_umbp_host_allocator.py`: host memory allocator paths, including anonymous / hugetlb / NUMA / prefault behavior against a fake backing.
- `test_umbp_store.py`: `UMBPStore` put/get behavior against a mocked `HostKVCache` with `page_first` layout, covering MHA and MLA cases.

## Known Limitations

1. `--hicache-mem-layout page_first` is required. `layer_first` currently asserts in the UMBP put path.
2. `--hicache-write-policy write_through` deadlocks the detokenizer on some launch configurations (including the DSR1-MXFP4 correctness launch above); prefer `write_back` unless your configuration has been validated end-to-end.
3. Carries the AITER MLA `--page-size > 1` correctness fix from #25556 as a cherry-pick; drop once #25556 lands upstream.

## Reproduction Commands

### Correctness test commands

Start `umbp_master` for the mori run:

```bash
nohup umbp_master 0.0.0.0:7777 > umbp_master.log 2>&1 &
```

Baseline server, without mori:

```bash
python3 -m sglang.launch_server \
  --model-path "$MODEL_PATH" \
  --served-model-name DSR1 --trust-remote-code \
  --host 0.0.0.0 --port 30100 \
  --tp 8 --dp 8 --ep-size 8 \
  --enable-dp-attention --enable-dp-lm-head \
  --page-size 64 --kv-cache-dtype fp8_e4m3 \
  --mem-fraction-static 0.7225 \
  --chunked-prefill-size 2048 --max-prefill-tokens 16384 \
  --schedule-conservativeness 0.3 --max-running-requests 64 \
  --attention-backend aiter \
  --disable-cuda-graph --disable-overlap-schedule --disable-piecewise-cuda-graph \
  --moe-a2a-backend mori --deepep-mode normal
```

mori server uses the same launch command with these flags appended:

```bash
--enable-hierarchical-cache --hicache-ratio 1.0 \
--hicache-write-policy write_back --hicache-mem-layout page_first \
--hicache-storage-backend mori \
--hicache-storage-backend-extra-config @mori_acc.json
```

`mori_acc.json`:

```json
{
  "dram_capacity_bytes": 17179869184,
  "ssd_enabled": false,
  "eviction_policy": "lru",
  "master_address": "127.0.0.1:7777",
  "node_address": "127.0.0.1"
}
```

Run lm-eval after the server is ready:

```bash
lm_eval --model local-completions \
  --model_args base_url=http://127.0.0.1:30100/v1/completions,model=DSR1,tokenizer=$MODEL_PATH,tokenized_requests=False,num_concurrent=64 \
  --tasks gsm8k --limit 200 --output_path lm_eval_run
```

`flexible-extract.exact_match` is in `lm_eval_run/DSR1/results_*.json`.

### Cache-hit / TTFT test commands

Case 2 and Case 3 require `umbp_master` to be running on `127.0.0.1:7777` (start it as in the correctness section).

Case 1, HBM only:

```bash
python -m sglang.launch_server \
  --enable-cache-report --enable-metrics \
  --model-path "$MODEL_PATH" --host 0.0.0.0 --port 30100 \
  --tp-size 8 --dp-size 8 --enable-dp-attention \
  --moe-a2a-backend mori --page-size 64 \
  --max-total-tokens 20000
```

Case 2, HBM + mori DRAM:

```bash
python -m sglang.launch_server \
  --enable-cache-report --enable-metrics \
  --model-path "$MODEL_PATH" --host 0.0.0.0 --port 30100 \
  --tp-size 8 --dp-size 8 --enable-dp-attention \
  --moe-a2a-backend mori --page-size 64 \
  --max-total-tokens 20000 \
  --enable-hierarchical-cache \
  --hicache-size 2 \
  --hicache-write-policy write_through --hicache-mem-layout page_first \
  --hicache-storage-backend mori \
  --hicache-storage-backend-extra-config '{"dram_capacity_bytes": 17179869184, "ssd_enabled": false, "auto_promote_on_read": true, "eviction_policy": "prefix_aware_lru", "master_address": "127.0.0.1:7777", "node_address": "127.0.0.1"}'
```

Case 3, HBM + mori DRAM + SSD config smoke:

```bash
python -m sglang.launch_server \
  --enable-cache-report --enable-metrics \
  --model-path "$MODEL_PATH" --host 0.0.0.0 --port 30100 \
  --tp-size 8 --dp-size 8 --enable-dp-attention \
  --moe-a2a-backend mori --page-size 64 \
  --max-total-tokens 20000 \
  --enable-hierarchical-cache \
  --hicache-size 2 \
  --hicache-write-policy write_through --hicache-mem-layout page_first \
  --hicache-storage-backend mori \
  --hicache-storage-backend-extra-config '{"dram_capacity_bytes": 17179869184, "ssd_enabled": false, "ssd_storage_dir": "/tmp/umbp_ssd", "ssd_capacity_bytes": 0, "auto_promote_on_read": true, "eviction_policy": "prefix_aware_lru", "master_address": "127.0.0.1:7777", "node_address": "127.0.0.1"}'
```

Run the workload after each server is ready:

```bash
python benchmark/hicache/bench_multiturn.py \
  --model-path "$MODEL_PATH" --port 30100 \
  --num-clients 64 --max-parallel 32 --num-rounds 20 \
  --request-length 2048 --sub-question-input-length 430 \
  --output-length 1 --request-rate 10 \
  --disable-auto-run --enable-round-barrier \
  --seed 42 --log-file ./case_metrics.jsonl --tag case
```

Per-case `cache_hit_rate`, `avg_ttft_ms`, and `p90_ttft_ms` are in the `summary` JSON line of `case_metrics.jsonl`.
