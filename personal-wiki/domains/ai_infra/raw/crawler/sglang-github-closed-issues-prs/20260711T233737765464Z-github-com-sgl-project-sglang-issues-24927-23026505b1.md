### Describe the bug

Loading a VLM checkpoint quantized via `compressed-tensors` (e.g., `RedHatAI/gemma-4-31B-it-NVFP4`, `cyankiwi/gemma-4-31B-it-AWQ-4bit`) crashes during warmup, or on the first image-containing request, with:

```
ValueError: mm_fp4 accepts 2d tensors, got torch.Size([1, 2520, 576]) and torch.Size([576, 3456])
  File ".../sglang/srt/layers/clippable_linear.py", line 157, in forward
  File ".../sglang/srt/layers/quantization/compressed_tensors/schemes/compressed_tensors_w4a4_nvfp4.py", line 160, in apply_weights
  File ".../flashinfer/gemm/gemm_base.py", line 4367, in _check_mm_fp4_problem_size
```

#### Root cause

These checkpoints follow the standard VLM quantization pattern: the **language model is quantized** while the **vision tower is kept in BF16** and listed in `quantization_config.ignore`. The ignore entries include a `.linear` submodule suffix (this comes from the `CompressedLinear` wrapper that llm-compressor applies), e.g.:

```jsonc
"ignore": [
  "model.vision_tower.encoder.layers.0.self_attn.q_proj.linear",
  "model.vision_tower.encoder.layers.0.self_attn.k_proj.linear",
  "model.vision_tower.encoder.layers.0.self_attn.v_proj.linear",
  // ... 192 entries total for Gemma-4 / cyankiwi
]
```

But SGLang's Gemma-4 model uses `packed_modules_mapping` to expand a fused `qkv_proj` into shard names without the `.linear` suffix:

```
runtime layer_name:  model.vision_tower.encoder.layers.0.self_attn.q_proj
ignore entry:        model.vision_tower.encoder.layers.0.self_attn.q_proj.linear
```

`compressed_tensors/utils.py::_is_equal_or_regex_match(check_contains=True)` only matches `target in value`. Here the target (with `.linear`) is **longer** than the value (without `.linear`), so the substring check fails one direction and the layer is **not** marked as ignored. The fused layer then gets the NVFP4 (or AWQ) quant scheme applied, and during forward the vision tower pushes 3D activations through `fp4_gemm`, which only accepts 2D tensors → crash.

vLLM avoids this entirely because it constructs the Gemma-4 vision tower via `transformers.AutoModel` (which has no awareness of vLLM's quant path); SGLang's native vision tower is the only one that exercises this code path with `.linear`-suffixed ignore entries.

### Reproduction

```bash
# venv: Python 3.12.13 + torch 2.11.0+cu130 + sglang 0.5.11 (source build of v0.5.11 tag)
python -m sglang.launch_server \
  --model-path RedHatAI/gemma-4-31B-it-NVFP4 \
  --attention-backend triton \
  --kv-cache-dtype fp8_e4m3 \
  --fp4-gemm-backend flashinfer_cutlass \
  --mem-fraction-static 0.85 \
  --context-length 8192 \
  --port 30000 \
  --trust-remote-code
```

The server crashes during warmup with the `mm_fp4 accepts 2d tensors` error above. Adding `--skip-server-warmup` works around the warmup crash but image requests still fail at the same point.

### Fix

The proposed fix (PR linked below) adds a path-extension match in `_is_equal_or_regex_match`: if `target` starts with `value + "."`, treat it as a match. The `.` separator prevents over-match (`q` does not match `q_proj.linear`).

After applying the patch:
- Warmup + CUDA-graph capture complete normally.
- `Gemma4ForConditionalGeneration` loads `vision_tower.*` as BF16 (no NVFP4 path).
- Text chat works: a 19-token haiku in ~0.44 s warm on a single RTX 5090.
- Image OCR works: a synthetic invoice PNG (7 lines) round-trips through the vision tower in ~2.3 s for 105 tokens.
