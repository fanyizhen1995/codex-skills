## What

Bump `TVM_FFI_VER` in `docker/sgl-deep-gemm.Dockerfile` `0.1.9` Ôćĺ `0.1.11`. This is the **build-time** `apache-tvm-ffi` that `release-whl-deepgemm.yml` installs to compile the `sgl-deep-gemm` wheel's `_C.so` against tvm-ffi's C++ ABI.

Runtime-safe on its own ÔÇö it changes nothing in the current install; it only affects what the *next* `sgl-deep-gemm` wheel build compiles against.

## Why

Companion to **sgl-project/DeepGEMM#47**, which bumps the wheel's *runtime* `apache-tvm-ffi` pin to `0.1.11` (`sgl_deep_gemm/pyproject.toml` + `setup.py`). Build-time and runtime tvm-ffi must agree, or the published wheel links one ABI and resolves another. `0.1.11` (not latest `0.1.12`) avoids the duplicate `__ffi_repr__` import-abort regression (tile-ai/tilelang#2367).

## Rollout sequence

1. **sgl-project/DeepGEMM#47** ÔÇö bump wheel runtime pin to 0.1.11 (Ôćĺ `dev`).
2. **This PR** ÔÇö bump build-time `TVM_FFI_VER` to 0.1.11.
3. **Release** ÔÇö run `release-whl-deepgemm.yml` to build + publish a new `sgl-deep-gemm` version (compiled against 0.1.11, from DeepGEMM `dev`).
4. **Follow-up PR (PR-B)** ÔÇö bump the SGLang *runtime* pins together (transitively coupled; needs the new wheel):
   - `python/pyproject.toml`: `apache-tvm-ffi 0.1.9 Ôćĺ 0.1.11`, `tilelang 0.1.8 Ôćĺ 0.1.11`, `sgl-deep-gemm 0.1.2 Ôćĺ <new>`
   - `docker/Dockerfile`: `SGL_DEEP_GEMM_VERSION 0.1.2 Ôćĺ <new>`

   PR-B can't land until the new wheel is on the index ÔÇö `sgl-deep-gemm 0.1.2` pins `apache-tvm-ffi==0.1.9`, so it can't co-resolve with `0.1.11`.
