# feat(sgl-model-gateway): add in-flight request cap (max_inflight_requests)

## What

Add a true in-flight request cap (`max_inflight_requests`) to `sgl-model-gateway`.

`max_concurrent_requests` behaves as a requests-per-second limiter (`TokenBucket::new(n, n)` with timer-based refill — tokens refill on a clock regardless of whether requests are still running). This complements it with a true semaphore that bounds the number of concurrently in-flight requests.

## How

New opt-in config `max_inflight_requests` (default `0` = disabled, `-1` = explicit disable). Existing deployments are unchanged.

- **Semaphore.** `AppContext.inflight_limiter` is `TokenBucket::new(cap, 0)` — `refill_rate = 0`, so permits return only when a response stream ends (via `TokenGuardBody`), not on a timer.
- **Gate order.** `concurrency_limit_middleware`: mesh global rate → local req/s bucket (+ optional queue) → in-flight cap. The in-flight permit is acquired *after* a req/s token is granted, so a request queued for a req/s token holds no in-flight slot. On in-flight overflow the request is rejected with `429` (not re-queued) and its req/s token is released.
- **RAII permit.** `InflightPermit` returns the permit on drop if the request future is dropped before the body guard is installed (client disconnect). `disarm()` hands ownership to `TokenGuardBody` for the normal path.
- **Surfaces.** `--max-inflight-requests` CLI flag (default `0`) and a Python binding keyword arg (default `0`).
